# BankBridge（汇桥）v0.3.0 架构设计：身份认证与权限控制

> 版本：v0.3.0  
> 目标：让系统具备基础的企业访问边界，实现 JWT 认证 + RBAC 权限矩阵 + 数据脱敏  
> 状态：设计中

---

## 1. 设计目标

v0.3.0 在 v0.2.0 的异步可靠性基础上，引入**安全边界**能力：

1. **身份认证**：所有 API 请求必须携带有效 JWT Token，未认证请求返回 `401`。
2. **权限控制**：基于角色的访问控制（RBAC），不同角色只能执行被授权的操作，越权返回 `403`。
3. **数据脱敏**：敏感字段（账户号、姓名等）在日志和未授权场景的 API 响应中脱敏输出。
4. **操作审计**：关键操作（登录、导出、规则修改）写入审计事件，支持追溯。

**不是目标**：
- 真实 OAuth2 授权服务器（如 Keycloak、Auth0）集成——保持项目独立可运行
- 完整的用户注册流程——通过 Flyway 预置种子用户
- TLS 证书管理——本地 Docker 使用 HTTP；共享部署必须由反向代理或网关终止 TLS

## 2. 核心概念：为什么用 JWT + RBAC？

### 2.1 JWT vs Session

| 维度 | Session（服务端状态） | JWT（无状态） |
|------|----------------------|--------------|
| 服务端存储 | 需要 Redis/内存存 Session | 不存，Token 自包含信息 |
| 水平扩展 | 需要共享 Session 存储 | 天然支持，任意实例可验证 |
| 过期控制 | 服务端主动删除 | 依赖 Token 的 `exp` 字段，不可提前失效 |
| 适用场景 | 需要服务端强制下线的场景 | 微服务、无状态 API、短时 Token |

**BankBridge 选择 JWT 的原因**：
- 项目是无状态 REST API，不需要服务端强制下线用户
- 未来向微服务演进时，JWT 可以在服务间传递用户上下文
- 教学场景下，JWT 的实现和原理更容易"讲清楚"

### 2.2 RBAC 模型

```
User --(1:N)--> Role --(N:M)--> Permission --(N:M)--> Resource/Operation
```

- **User**：系统用户，通过用户名/密码登录
- **Role**：角色，如 `OPERATOR`、`COMPLIANCE_ANALYST`
- **Permission**：细粒度权限，如 `PAYMENT_CREATE`、`RULE_UPDATE`
- **Resource/Operation**：受保护的 API 端点

**为什么用 RBAC 而不是 ACL？**
- RBAC 以角色为中介，管理复杂度 O(R) 而不是 O(U×R)
- 企业系统普遍采用 RBAC，设计分析中认知度高
- 教学场景下，四角色 × 五权限的矩阵清晰易懂

## 3. 认证架构：JWT 流程

```
┌─────────┐   POST /api/auth/login   ┌─────────┐
│ Client  │ ────────────────────────> │  Auth   │
│         │  {username, password}     │Controller│
│         │ <──────────────────────── │         │
│         │  {accessToken, tokenType,│         │
│         │   expiresIn}              │         │
└─────────┘                           └─────────┘
                                              │
                                              │ 验证密码
                                              │ 生成 JWT
                                              ▼
                                        ┌─────────┐
                                        │ UserDetails│
                                        │Service  │
                                        └─────────┘

后续请求：
┌─────────┐   GET /api/payments        ┌─────────┐
│ Client  │ ────────────────────────> │  API    │
│         │  Authorization: Bearer    │Controller│
│         │  <token>                  │         │
│         │ <──────────────────────── │         │
│         │  401/403 or 200 + data    │         │
└─────────┘                           └─────────┘
                                              │
                                              │ Spring Security
                                              │ JWT 过滤器验证
                                              ▼
                                        ┌─────────┐
                                        │JwtAuth  │
                                        │Filter   │
                                        └─────────┘
```

### 3.1 Token 结构（JWS）

```json
// Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload
{
  "sub": "operator001",      // 用户名
  "roles": ["OPERATOR"],     // 角色数组
  "iat": 1720000000,          // 签发时间
  "exp": 1720003600           // 过期时间（1小时）
}

// Signature: HMACSHA256(base64(header) + "." + base64(payload), secret)
```

### 3.2 密码安全

- 存储：使用 BCrypt 哈希，强度因子 12（默认）
- 传输：Login 请求走 HTTPS（Docker 内配置）
- 不存储原始密码，连 seed 数据中的密码也是 BCrypt 哈希后的值

## 4. 授权架构：RBAC 权限矩阵

### 4.1 角色定义

| 角色 | 职责 | 典型用户 |
|------|------|----------|
| `OPERATOR` | 日常操作：提交付款、查询状态、批量处理 | 柜员、操作员 |
| `COMPLIANCE_ANALYST` | 合规审查：处理人工审核、查看筛查结果 | 合规专员 |
| `AUDITOR` | 审计追溯：查看审计事件、对账记录、导出报表 | 内审员 |
| `ADMIN` | 系统管理：维护规则、重放死信、管理用户 | 系统管理员 |

### 4.2 权限清单（Permission）

```java
public enum Permission {
    // 付款操作
    PAYMENT_CREATE,    // 提交付款
    PAYMENT_READ,      // 查询付款
    PAYMENT_ADMIN,     // 所有付款操作（含删除/修改）
    
    // 合规操作
    SCREENING_REVIEW,  // 人工审核
    RULE_READ,         // 查看规则
    RULE_WRITE,        // 创建/修改/删除规则
    
    // 审计操作
    AUDIT_READ,        // 查看审计事件
    REPORT_EXPORT,     // 导出报表
    
    // 系统操作
    DEADLETTER_REPLAY, // 重放死信
    ADMIN_OPERATIONS   // 所有管理操作
}
```

### 4.3 权限矩阵

| 操作 | OPERATOR | COMPLIANCE_ANALYST | AUDITOR | ADMIN |
|------|:--------:|:------------------:|:-------:|:-----:|
| `POST /api/payments` | ✅ | ❌ | ❌ | ✅ |
| `GET /api/payments` | ✅ | ✅ | ✅ | ✅ |
| `PUT /api/payments/{id}/manual-review` | ❌ | ✅ | ❌ | ✅ |
| `GET /api/screening-rules` | ✅ | ✅ | ✅ | ✅ |
| `POST /api/screening-rules` | ❌ | ❌ | ❌ | ✅ |
| `GET /api/audit-events` | ❌ | 受限* | ✅ | ✅ |
| `POST /api/dead-letter/replay` | ❌ | ❌ | ❌ | ✅ |
| `POST /api/reports/export` | ❌ | ❌ | ✅ | ✅ |
| `POST /api/batch/upload` | ✅ | ❌ | ❌ | ✅ |

*受限：COMPLIANCE_ANALYST 只能查看自己处理的审核相关审计事件

### 4.4 方法级安全

使用 Spring Security 的 `@PreAuthorize`：

```java
@PreAuthorize("hasAnyRole('OPERATOR', 'ADMIN')")
@PostMapping("/api/payments")
public PaymentResponse create(@RequestBody @Valid CreatePaymentRequest request) { ... }

@PreAuthorize("hasAnyRole('COMPLIANCE_ANALYST', 'ADMIN')")
@PutMapping("/api/payments/{id}/manual-review")
public PaymentResponse manualReview(...) { ... }

@PreAuthorize("hasRole('ADMIN')")
@PostMapping("/api/dead-letter/replay")
public DeadLetterReplayResponse replay(...) { ... }

@PreAuthorize("hasAnyRole('AUDITOR', 'ADMIN')")
@PostMapping("/api/reports/export")
public ResponseEntity<Resource> exportReport(...) { ... }
```

## 5. 数据脱敏策略

### 5.1 脱敏规则

| 字段 | 全可见角色 | 脱敏角色 | 脱敏方式 | 示例 |
|------|-----------|----------|----------|------|
| `debtorAccount` | ADMIN | 其他 | 保留前4后4 | `GB12****5678` |
| `creditorAccount` | ADMIN | 其他 | 保留前4后4 | `FR76****9012` |
| `debtorName` | ADMIN | 其他 | 保留姓 + * | `张**` |
| `creditorName` | ADMIN | 其他 | 保留姓 + * | `L****` |
| `amount` | 全部 | — | 无脱敏 | — |
| `screeningRule.matchValue` | ADMIN | 其他 | 完整隐藏 | `***` |

### 5.2 脱敏实现方式

**方案 A：DTO 层脱敏（推荐）**

在 `PaymentResponse` 的构建过程中，根据当前用户角色决定是否脱敏：

```java
public PaymentResponse fromEntity(PaymentInstruction payment, UserRole viewerRole) {
    String debtorAccount = viewerRole == ADMIN 
        ? payment.getDebtorAccount() 
        : maskAccount(payment.getDebtorAccount());
    // ...
}
```

**方案 B：JSON 序列化器脱敏**

自定义 Jackson 序列化器，在序列化时根据 SecurityContext 中的角色脱敏。

**选择方案 A 的原因**：
- 显式控制，代码可读性好，教学场景下更容易理解
- 不依赖 Jackson 的隐式行为，调试更直观

### 5.3 日志脱敏

```java
// 脱敏前：log.info("Processing payment from {} to {}", debtorAccount, creditorAccount);
// 脱敏后：log.info("Processing payment from {} to {}", maskAccount(debtorAccount), maskAccount(creditorAccount));
```

使用 `@ToString.Exclude` 或 `@JsonIgnore` 防止敏感字段意外出现在日志中。

## 9. 设计分析高频问答（v0.3.0 新增）

### Q1: 为什么用 JWT 而不是 Session？

> **30 秒版：** 项目是无状态 REST API，JWT 不需要服务端存储 Session，天然支持水平扩展。教学场景下实现和原理更容易讲清楚。
>
> **3 分钟深挖：** 
> - JWT 是自包含的，Token 里携带了用户身份和角色，服务端不需要查库就能验证
> - 适合微服务架构，服务间可以透传 Token 来传递用户上下文
> - 缺点是不可提前失效（除非引入黑名单或缩短过期时间），对于需要强制下线的场景不如 Session 灵活
> - BankBridge 是教学沙盒，不需要强制下线，所以用 JWT 更合适

### Q2: RBAC 和 ABAC 有什么区别？为什么选 RBAC？

> **答：** RBAC（Role-Based）以角色为权限的载体，ABAC（Attribute-Based）以属性（用户部门、时间、IP 等）动态计算权限。
> - RBAC 简单、固定，适合权限变化不频繁的企业系统，管理复杂度是 O(R)
> - ABAC 灵活、动态，适合复杂的多租户 SaaS，但实现和调试成本高
> - BankBridge 是四角色的固定权限矩阵，RBAC 足够清晰，设计分析中认知度高，容易讲清楚

### Q3: 方法级安全（@PreAuthorize）和 URL 级安全（SecurityFilterChain）怎么配合？

> **答：** 两者互补：
> - `SecurityFilterChain` 做粗粒度控制：哪些 URL 需要认证、哪些公开
> - `@PreAuthorize` 做细粒度控制：具体到方法的权限检查，比如 `POST /api/payments` 需要 OPERATOR 或 ADMIN
> - 粗粒度在前，细粒度在后，形成「先过滤、再校验」的层次化安全

### Q4: 数据脱敏放在哪一层？为什么？

> **答：** 放在 DTO 层（Presentation Layer）。
> - 不在数据库层脱敏：数据库需要存完整数据，脱敏后无法对账
> - 不在 Entity 层脱敏：Entity 是领域模型，不应该知道当前用户是谁
> - DTO 层负责对外表示，根据当前用户角色决定返回完整还是脱敏数据，职责清晰
> - 教学场景下代码显式，容易理解

### Q5: 如果 Token 泄露了，怎么减轻风险？

> **答：** 多层防御：
> 1. **短时效**：Token 有效期 1 小时，泄露窗口有限
> 2. **HTTPS 传输**：防止中间人截获
> 3. **敏感操作二次确认**：如导出报表、重放死信需要额外的操作确认（v0.3.0 简化版）
> 4. **审计日志**：所有操作记录到 `audit_events`，泄露后可追溯
> 5. **生产环境可升级**：引入 Refresh Token 轮换、Token 黑名单（Redis）或直接使用 OAuth2 授权服务器

### Q6: 测试登录流程时，怎么避免在代码里写明文密码？

> **答：** 测试中使用 Testcontainers 或 H2 的 seed 数据，密码是 BCrypt 哈希后的值。测试中通过 `loginAs()` 辅助方法获取 Token，密码硬编码在测试类常量中（教学场景下可接受）。生产环境中应使用环境变量或 secrets manager。
